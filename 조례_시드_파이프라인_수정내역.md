## 수정 1. `extract_doc_meta()` 배선 (발제 B-4)

### 문제

`statute_parser.py`에 `extract_doc_meta()`가 구현돼 있는데 **아무도 호출하지 않았습니다.**
`ingest_statutes.py`는 `parse_statute, length_report`만 import 했습니다.

또 `parse_statute()`가 `document_id` / `district_id`를 키워드 인자로 받는데 ingest가 넘기지 않아,
`None`이면 메타에서 제외되는 로직 때문에 **전 청크에서 누락**됐습니다.

```
document_id 있는 청크: 0
district_id 있는 청크: 0
```

발제 B-4의 목적이 "조례 개정 시 **문서 단위** 삭제·재적재"인데, 문서를 식별할 키가 하나도 없으면
여전히 콜렉션 전체 재적재밖에 못 합니다. 시드가 1건이라 지금은 티가 안 나지만 2건째부터 막힙니다.

### 수정

**`statute_parser.py`** — `parse_statute()`에 `doc_meta` 파라미터 추가

```python
def parse_statute(
    text, doc_title, *,
    facility_type="흡연부스",
    document_id=None,
    district_id=None,
    doc_meta=None,          # ← 추가
):
```

`base_meta()`에서 `doc_meta`의 키를 전 청크 메타데이터에 병합합니다
(`doc_title_detected`는 `document_title`과 중복이라 제외).

**`ingest_statutes.py`** — `extract_doc_meta()` 호출 후 전달

```python
doc_meta = extract_doc_meta(text)
chunks = parse_statute(text, title, facility_type=SEED_FACILITY_TYPE, doc_meta=doc_meta)
```

### 결과

```
메타 키: ['article_no', 'article_title', 'chapter', 'clause_no',
          'doc_no', 'document_title', 'enforcement_date', 'facility_type', 'source']
샘플: {'facility_type': '흡연부스', 'enforcement_date': '2014-12-26', 'doc_no': '제1081호'}
```

`enforcement_date` + `doc_no` 조합으로 **"어느 판(版)의 조문인지" 식별이 가능**해졌습니다.
조례 개정 시 이 값으로 구판을 걸러내 삭제할 수 있습니다.

### 남은 것

`document_id`(정수 PK)는 발제 #5의 `ordinance_documents` 테이블이 아직 없어서 **의도적으로 비워뒀습니다.**
테이블이 생기면 INSERT 후 받은 PK를 `parse_statute(document_id=...)`로 넘기면 됩니다.
`extract_doc_meta()`가 뽑는 시행일·조례번호가 그 테이블의 INSERT 재료입니다.

`district_id`도 자치구 코드 체계가 확정되면 같은 방식으로 넘깁니다.

## 수정 2. 검증 질의에 필터 적용

### 문제

적재 후 검증 질의가 `retrieve_similar_statutes(q, top_k=3)` — **`facility_type`을 넘기지 않았습니다.**

그런데 실제 서비스([`simulations.py`](app/api/v1/simulations.py))는 **항상** 필터를 겁니다.
즉 적재 태그와 검색 필터 값이 어긋나 있어도 **검증은 초록불, 서비스만 0건**이 되는 구조였습니다.
가장 잡기 어려운 종류의 버그입니다.

### 수정

검증을 **두 벌** 돌리도록 바꿨습니다.

1. **필터 없음** — 적재·임베딩 자체가 정상인지 확인
2. **필터 적용** (`facility_type=SEED_FACILITY_TYPE`) — 실제 서비스 경로 재현

필터 적용 질의가 **전부 0건이면 경고를 출력**합니다.

```python
if empty == len(TEST_QUERIES):
    print("⚠️ 필터 적용 시 전 질의 0건 — 적재 태그와 검색 필터 값이 어긋났을 가능성이 큽니다.")
```

두 벌의 결과가 갈리면 그 자체가 태그 불일치의 증거입니다.

## `facility_type` 값 조사 결과 — 적재 태그 `"흡연부스"`가 맞음

전 원격 브랜치를 훑어 실제 사용값을 전수 집계한 결과, **규민님 파서 기본값 `"흡연부스"`가 정확합니다.**
별도 조치가 필요 없습니다.

> 초기 검토에서 "적재 태그와 프론트 값이 어긋난 것 같다"고 지적했으나 **오진이었습니다.**
> `조례데이터.csv`의 `"스마트흡연부스"`를 API 필드로 오인한 것이며, 실제로는 다른 네임스페이스입니다.

### 왜 헷갈렸나 — 같은 이름의 필드가 3개 있음

`facility_type`이라는 이름이 **의미가 다른 세 곳**에서 쓰이고 있습니다. 값이 충돌한 게 아니라 용도가 다릅니다.

| # | 위치 | 의미 | 실제 값 (전 브랜치 집계) |
|---|---|---|---|
| **①** | `StreamRequest.facility_type` (요청 최상위) | **건립할** 시설 | `"흡연부스"` (18회) |
| ② | `audit_data.results[].roles[].facility_type` (중첩) | 배제 근거가 되는 **주변** 시설 | `"버스정류소"`(16) `"학교"`(8) `"지하철역"`(8) `"어린이집"`(8) `"어린이보호구역"`(8) |
| ③ | `smoking_area_polygons` 테이블 컬럼 | 흡연구역 폴리곤의 시설종류 | `"스마트흡연부스"` (1) |

RAG 필터가 받는 것은 **①번**입니다.
`simulations.py`가 `StreamRequest.facility_type`을 그대로 `retrieve_similar_statutes(facility_type=...)`에 넘깁니다.

②는 `audit_data` **안에 중첩된** 별개 필드이고,
③은 `load_data.py`가 CSV의 `"시설종류"` 컬럼을 DB 컬럼명 `facility_type`으로 매핑한 것입니다 — API와 무관합니다.

### 근거 — 독립된 두 소스가 일치

```python
# test_api_client.py:16  (실제 API 호출 클라이언트)
payload = {"parcel_id": 1, "facility_type": "흡연부스", "audit_data": audit_data}

# run_ai_console.py:107  (DB 없이 도는 콘솔 테스트)
"facility_type": "흡연부스",
```

### 남은 확인 (권장, 차단 사항 아님)

위 두 소스는 모두 **백엔드 쪽 테스트 코드**입니다. 프론트엔드가 다른 문자열을 보낼 여지는 이론상 남아 있습니다.
**장천명님(풀스택)께** `POST /api/v1/simulation/stream` 요청의 `facility_type` 실제 값을 한 번 크로스체크하면 확정됩니다.
다만 현재 근거로 `"흡연부스"` 진행에 문제 없으며, 실적재를 막을 사유는 아닙니다.

### 🔴 진짜 위험 — 이름 충돌 자체

값은 맞지만 **구조가 위험합니다.** 정반대 의미의 필드가 같은 이름으로 **한 요청 안에 공존**합니다.
`_parse_audit_data()`가 ②를 다루는 바로 옆에서 ①이 RAG 필터로 나갑니다.

나중에 누군가 "`facility_type` 표기를 통일하자"며 ②의 값(`"학교"` 등)을 ①에 대입하면
**RAG가 조용히 전량 0건**이 됩니다. 정확일치 필터라 예외도 안 나고 로그도 안 남으며,
토론은 "관련 조례 없음"으로 정상 진행되는 것처럼 보입니다.

권장 대응 (별도 이슈):

1. ①을 `Enum`으로 좁혀 오타·오대입을 타입 레벨에서 차단
2. 또는 ②를 `constraint_facility_type` 등으로 개명해 이름 충돌 자체를 제거

프론트 API 계약 변경이 필요해 이번 범위 밖으로 두고 별도 발제합니다.
